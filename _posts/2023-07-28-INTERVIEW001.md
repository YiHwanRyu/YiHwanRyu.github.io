## ◆ 예상질문-내용

### 1. NoSQL과 RDBMS의 특징과 차이점에 대해서 장, 단점을 들어 설명해주세요. (7/28)

- RDBMS는 관계형 모델에 따라 데이터를 구조화하고 저장하는 데이터베이스 관리 시스템이며, SQL 구문을 사용하여 데이터의 저장, 수정, 삭제를 할 수 있습니다.<br>
또한, 릴레이션(스키마 + 인스턴스)을 통해 데이터를 관리하기 때문에 구조화가 중요하고, 외래키를 통해 테이블간 조인이 가능한 것이 특징입니다. 

- NoSQL은 RDBMS와 달리 테이블간의 관계를 정의하지 않고, 비교적 자유로운 형태로 데이터를 저장하는 특징이 있습니다.<br>
따라서 수평확장에 RDBMS보다 유용하고, 단순히 구간을 나누는 것을 통해 분산처리가 가능합니다. <br>
(대표적 예 key-value store, bigtable)

- 하지만 NoSql은 명확한 데이터 구조를 정의하지 않고, 외래키가 없기 때문에 조인을 할 수 없고 중복처리가 어렵습니다. <br>
따라서 두 데이터베이스 관리형태는 각각 적절한 상황에 맞춰 사용해야 합니다.

### 2. mvc 패턴에 대해서 설명해주세요. (7/29)

- MVC 패턴은 Model, View, Controller로 구성된 디자인 패턴입니다.<br>
각각 Model은 데이터를 가진 객체, View는 화면에 표시되는 글자 등의 시각적요소, Controller는 비즈니스 로직이 구현된 모델과 뷰의 연결 역할을 수행합니다.

- MVC 패턴은 각각 구성요소의 역할을 분리하여 중복 개발되는 부분을 방지하고 효율적인 개발과 유지보수를 가능하게 합니다.

- 하지만 Model과 View사이의 의존성이 높다는 특징으로 인해 어플리케이션의 규모가 커질수록 구조가 복잡해지고, 유지보수가 어렵습니다.

### 3. ORM을 사용하면서 쿼리가 복잡해지는 경우에는 어떻게 해결하는게 좋을까요? (7/30)

- ORM은 Object Relation Mapping의 약어로, 객체와 DB를 각각 설계하면 객체가 DB의 테이블이 되도록 Mapping 시켜주는 기술입니다.

- Java에서 ORM은 JPA(Java Persistent Api), 자바 ORM Api 표준 명세를 통해 사용되고, 주로 구현체인 Hibernate로 사용됩니다.

- 이러한 ORM을 사용할 때, 쿼리가 복잡해진다면 단순히 표현하기가 어렵고, 성능이 raw query보다 느리다는 단점이 있습니다.

- 그래서 보통 복잡한 쿼리를 처리할 때는 JPQL이나 QueryDsl을 통해 쿼리문을 구성하여 해결할 수 있습니다.

### 4. RDBMS의 정규화에 대해 설명해주세요. (8/01)

- RDBMS의 정규화는 데이터 무결성을 유지하기 위해 구조화하여 중복을 최소화하는 것입니다. 즉, 테이블을 분리하는 과정이며, 1NF부터 BCNF까지 단계별로 정규화 단계가 존재합니다.

- 이를 통해 데이터베이스 변경 시 이상현상(삽입이상, 삭제이상, 갱신이상)을 제거할 수 있지만,

- 릴레이션을 나누어 구조화하기 때문에 join 등의 릴레이션간 연산이 증가하여 쿼리 응답시간이 늦어질 수 있습니다.

- 따라서 적절한 정규화를 통해 데이터베이스를 구조화하고, 필요에 따라 반대개념인 비정규화를 적용할 필요가 있습니다.

### 5. Primary Key, Foreign Key에 대해 설명해주세요. (8/02)

- Primary Key는 릴레이션을 대표하는 키로 테이블에 1개만 존재합니다. 또한, unique하며 null값을 가질 수 없는 키입니다.

- Foreign Key는 테이블 간의 관계를 설정하는 다른 테이블의 기본키입니다. 한 테이블에 여러 개가 존재할 수 있으며, Null값과 중복값을 가질 수 있습니다. 

- 두 키 모두 관계형데이터베이스에서 테이블 간의 관계를 설정하는 데 사용되며, FK 설정 시 다른 테이블의 기본키를 참조하기 때문에 데이터 무결성 유지에 도움이 됩니다.

### 6. HTTP 메서드에 대해 설명해주세요.(8/03)

- 클라이언트가 서버와 통신할 때 요청과 응답을 수행하는 방식입니다. 주로 클라이언트가 서버로 특정 리소스에 대하여 원하는 동작을 지정하여 요청하는 방식을 의미합니다.

- HTTP 메서드에는 GET, POST, PUT, PATCH, DELETE 등이 있으며, 일반적으로 데이터 조회에는 GET요청, 데이터의 등록에는 POST요청 그리고 삭제에는 DELETE요청을 서버로 보냅니다. 그리고 PUT과 PATCH는 데이터 수정 시 사용되나 PUT은 리소스의 대체 혹은 생성, PATCH는 부분변경을 의미하므로 사용 시 유의해야 합니다.

- 결과적으로 HTTP 메서드는 각각의 메서드에 특정한 목적이 정해져 있으므로 Restful API 설계에서 주요 규칙을 수행합니다. 

### 7. CORS(Cross Origin Resource Sharing)에 대해 설명해주세요.(8/4)

- CORS는 교차 출처 리소스 공유로 웹어플리케이션에서 서로 다른 출처 간의 리소스 요청이 가능하도록 권한을 부여하고, 브라우저에 알려주는 것입니다.

- 이러한 CORS 개념은 다른 출처의 리소스 접근 시 필수적입니다. 왜냐하면 기본적으로 브라우저의 보안정책이 SOP, same origin policy를 따르기 때문입니다.

### 8. 브라우저의 작동방식에 대해서 설명해주세요.(8/5)

- 브라우저는 사용자가 웹 컨텐츠에 접근할 수 있도록 돕는 소프트웨어 어플리케이션입니다. 

- 작동방식은 먼저 DNS를 통해 서버의 IP주소를 가져와 요청을 보냅니다. 그리고 웹서버는 해당 요청을 처리하고 HTML, CSS, JS 등의 리소스를 응답으로 보내줍니다.

- 결과적으로 이러한 과정을 통해 브라우저는 사용자에게 웹 페이지를 랜더링하고 상호작용할 수 있는 환경을 제공합니다.

### 9. 대용량 트래픽 발생 시 어떻게 대응해야 하나요?(8/7)

- 대용량 트래픽을 처리하기 위해서는 기존의 서버 리소스를 늘려야 합니다.

- 서버 리소스를 늘리는 방법에는 기존 서버의 성능을 높이는 수직확장과 동일한 목적의 서버를 여러대 추가시키는 수평확장이 있습니다.

- 수직확장은 성능 향상의 한계가 존재하기 때문에, 단일서버로 대용량 트래픽를 해결하는 것은 매우 어렵습니다. 따라서 로드밸런서를 통한 수평확장과 조합하여 비용 최적화를 통해 대용량 트래픽을 처리해야 합니다.

### 10. 쿠키, 세션의 개념과 차이를 설명해보세요(8/8)

- 쿠키와 세션은 웹 개발에서 사용되는 데이터 관리 방법입니다. 또한 두 방법은 웹이 무상태 프로토콜인 HTTP를 사용하기 때문에 필요합니다.

- 먼저, 쿠키는 클라이언트 측에 작은 데이터 파일을 저장하는 것이며, 세션은 서버에 데이터를 저장하여 사용자 정보를 유지하는 것입니다.

- 쿠키는 크기에 제한이 있으며 보안에 취약합니다. 하지만 개인화 설정에 적합합니다.

- 세션은 고유 식별자를 사용해 서버에 데이터를 저장하므로 보안이 강화되어 인증과 권한 부여에 주로 사용됩니다. 

### 11. TCP/UDP 에 대해서 설명해주세요(8/9)

- TCP(Transmission Control Protocol)은 신뢰성 있는 연결 지향형 통신 프로토콜로, 3-way handshaking을 통해 데이터 정확성과 순서를 보장하며, 웹 브라우징 및 파일 전송과 같은 안정적인 어플리케이션에 사용됩니다. 

- UDP(User Datagram Protocol)는 더 빠른 비연결형 프로토콜로, 확인 응답이나 순서가 없어 온라인 게임 및 스트리밍과 같은 빠른 전송이 필요한 어플리케이션에 적합합니다.

### 12. http, https 차이점에 대해 설명해주세요 (8/10)

- http는 인터넷을 통해 데이터를 전송하기 위해 사용되는 표준 프로토콜입니다. 기본적으로 80번 포트를 이용하고, 평문으로 데이터를 송수신하기 때문에 보안에 취약한 단점이 있습니다.

- 하지만 https는 SSL (Secure Sockets Layer) 또는 그 후속 버전인 TLS (Transport Layer Security)을 사용하여 클라이언트와 서버 간에 암호화된 연결을 설정하기 때문에 보안수준이 상대적으로 높습니다. 또한 기본적으로 443번 포트를 사용합니다.

### 13. DI, IoC에 대해 설명해주세요. (8/11)

- 의존성 주입(Dependency Injection, DI)은 소프트웨어 디자인 패턴으로, 컴포넌트의 의존성을 외부에서 주입하여 모듈 간 결합도를 낮추고 유연성을 향상시킵니다. 또한 테스트 용이성과 유지보수성을 증가 시킵니다.

- 제어의 역전(Inversion of Control, IoC)은 프로그램의 제어 흐름을 외부(프레임워크 등)에 위임하여 유지보수 및 확장성을 증진시키는 개념입니다.

- DI는 IoC의 구체적인 구현 방법 중 하나로, 애플리케이션의 전반적인 아키텍처 개선을 지원합니다.

### 14. 객체지향 프로그래밍이란 무엇이고 어떻게 활용할 수 있나요?(8/12)

- 객체지향 프로그래밍은 데이터와 기능을 객체로 구성하여 모듈화하고 재사용성을 높이는 방식입니다.

- 또한 각 객체의 의존성을 분리하여 변경영향을 최소화하고, 독립적인 개발을 가능하게 합니다.

- 따라서 대형 프로젝트에서 협업에 용이하고 유지보수에 효율적입니다.

### 15. TDD에 대해서 설명해주세요.(8/14)

- 테스트 주도 개발(TDD)은 실제 코드를 작성하기 전에 테스트를 먼저 작성하는 소프트웨어 개발 방법론입니다. 

- 이러한 TDD는 실패 테스트 작성, 해당 실패 테스트를 통과할 최소한의 코드작성, 리팩토링 순서로 개발이 진행됩니다.

- 따라서 TDD는 안정성과 코드 품질을 높이고 버그를 빠르게 감지할 수 있는 장점이 있습니다.

- 하지만 개발 시간의 증가와 초기 학습 및 테스트 유지 관리가 어렵다는 단점이 있습니다.

### 16. GET, POST의 개념과 함께 데이터 흐름에 대해서 설명해주세요. (8/15)

- GET은 서버에 데이터를 요청하는 데 사용합니다. 클라이언트가 GET 요청을 보낼 때 일반적으로 데이터는 URL의 쿼리 매개변수에 포함되며, 이러한 매개변수는 URL 자체에서 볼 수 있으므로 북마크하거나 공유할 수 있습니다. 또한, GET 요청은 서버의 상태를 변경시키지 않기 때문에 멱등적입니다.

- POST는 일반적으로 양식 제출, 파일 업로드 또는 기타 복잡한 데이터를 서버에 보내는 데 사용합니다. 이 때, 데이터는 URL에 표시되지 않는 Request Body에 포함됩니다. 이러한 POST 요청은 생성, 업데이트와 같은 멱등성이 유지되지 않는 작업을 수행합니다.

### 17. OSI 7계층에 대해 아는대로 설명해주세요.(8/16)

- OSI 7계층은 국제 표준화 단체인 ISO에서 제안한 네트워크 시스템 모델입니다. 각 계층은 고유의 기능을 수행하고, 하위의 계층이 바로 위의 계층으로 서비스를 제공합니다.

- 물리, 데이터 링크, 네트워크, 전송, 세션, 표현, 응용계층으로 구성되어 있으며, 이러한 모델의 가장 큰 도입 이유는 통신 과정의 단계적 파악을 통한 문제해결의 용이성입니다. 

- 각각의 설명을 요청한다면?
  - 1계층은 물리 계층으로, 전기적특성을 이용해서 통신 케이블로 데이터를 전송합니다.
  - 2계층은 데이터링크 계층으로, 스위치를 통해 MAC주소를 이용하여 물리계층에서 송수신되는 정보의 오류 검출 및 흐름을 관리합니다.(오류 제어는 재전송 방법으로 처리)
  - 3계층은 네트워크 계층으로, 라우터(Router)를 통해 경로를 선택하고 주소(IP)를 정하고 경로(Route)에 따라 패킷을 전달합니다.
  - 4계층은 전송 계층으로, TCP/UDP 프로토콜을 사용하여 종단간(end-to-end) 신뢰성 있고 효율적인 데이터를 전송하며, 오류검출 및 복구와 흐름제어, 중복검사 등을 수행합니다.
  - 5계층은 세션 계층으로, 사용자간의 포트 연결(session)이 유효한지 확인하고 설정합니다.
  - 6계층은 표현 계층으로, 데이터 표현이 상이한 응용 프로세스의 독립성을 제공하고, 암호화합니다.
  - 7계층은 응용 계층으로, 응용 프로세스와 직접 관계하여 일반적인 응용 서비스를 수행합니다.(UI)

### 18. 세션 기반 인증과 토큰 기반 인증의 차이에 대해 설명해주세요.(8/17)

- 세션 기반 인증은 사용자 인증을 위해 서버 측에서 사용자 정보를 저장 및 유지하는 방식입니다. 따라서 Stateful 한 접근 방식이며, 세션 하이재킹 및 CSRF에 취약하기 때문에 안전한 세션 관리가 필요합니다. 또한, 추적을 위해 세션 식별자 쿠키를 사용합니다.

- 토큰 기반 인증은 사용자 인증을 위해 JWT와 같은 토큰을 활용하는 방식입니다.
토큰을 통해 사용자의 정보를 전달하기 때문에 서버에 사용자 상태를 저장하지 않습니다. 따라서 Stateless 하기 때문에 확장성이 높고, 세션 하이재킹과 CSRF위험을 줄일 수 있습니다. 하지만 적절한 토큰 유효성 검사가 필요합니다.

### 19. JWT, Refresh, Access Token에 대해서 설명해주세요. (8/18)

- JWT는 JSON Web Token의 약어로 헤더, 페이로드, 서명으로 구성된 JSON 형식의 토큰입니다. 무상태이고 필요정보를 모두 토큰에 담고 있기 때문에 탈 중앙화가 가능합니다. 따라서 높은 유연성과 무결성으로 인해 인증 및 인가 프로세스에 많이 사용되고 있습니다. 

- 이러한 JWT는 서버 측에서 즉시 회수가 불가능하여, 탈취 시에 토큰 만료기한까지 다른 사용자가 이용할 수도 있기 때문에 Access token와 Refresh Token으로 나뉘어 사용됩니다.

- Access token은 실제 인증 프로세스에 사용되는 JWT이며, 짧은 만료 기한을 주고 세밀한 제어가 필요한 토큰입니다.

- Refresh token은 상대적으로 사용 가능 기한이 긴 토큰으로 오직 Access token의 재발급에만 관여하도록 범위를 지정한 토큰입니다.

- 이렇게 용도에 따라 토큰을 생성하여 보안을 강화할 수 있으나 Refresh token은 서버에 저장이 필요하고, 두 토큰의 검증 로직이 복잡해지는 단점이 있습니다.

### 20. OAuth에 대해서 설명해주세요. (8/19)

- OAuth는 Open Authorization의 약어로 접근 위임의 개방형 표준입니다. 접근 위임이란 사용자들이 비밀번호를 제공하지 않고, 다른 웹사이트에 자신들의 정보 접근 권한을 부여하는 것을 말합니다.
- OAuth는 인증의 주체인 Resource Owner, 권한을 위임받는 주체인 Client, 인증과 권한을 부여하는 Authorization Server, 그리고 사용자의 정보를 제공하는 Resource Server로 구성되어 있습니다.  
- 이러한 OAuth는 신뢰성 있는 인가 서버 사용을 통해 보안성을 향상시킬 수 있으며, 토큰 기반의 접근 방식을 통해 확장성을 높일 수 있습니다. 하지만 복잡한 처리로직과 인가 서버에 높은 의존성을 가지는 단점이 있습니다.

### 21. 쿼리 최적화에 대해 설명해주시고 방법에 대해 설명해주세요. (8/21)

- 쿼리 최적화는 데이터베이스 쿼리의 효율성과 성능을 향상시키는 과정입니다.

- 쿼리 최적화에는 자주 사용되는 열에 인덱스를 생성하여 검색 속도를 향상시키는 인덱싱, 자주 접근하는 데이터를 임시 저장하여 활용하는 캐싱, 적절한 조인을 위한 비정규화 등이 있습니다.

- 결론적으로 쿼리 최적화는 현재 사용되는 쿼리와 데이터베이스의 스키마를 분석하여 더 적은 리소스 사용으로 데이터 검색 속도를 향상시키는 것을 의미합니다.


### 22. 클래스형과 함수형의 차이를 설명해주세요. 어떤 방식을 주로 사용하였고 그 이유가 뭔지 답변해주세요. (8/22)

- 클래스형과 함수형은 프로그래밍 패러다임의 한 종류입니다. 클래스형(객체지향)은 객체를 중심으로 프로그램을 작성하는 패러다임이고, 함수형은 함수를 중심으로 프로그램을 작성하는 패러다임입니다.

- 클래스형 프로그래밍은 객체를 중심으로 프로그램을 작성하기 때문에 복잡한 프로그램을 작성하기 쉽고, 상태를 가지기 때문에 데이터를 저장하고 관리하기 쉽습니다. 함수형 프로그래밍은 함수를 중심으로 프로그램을 작성하기 때문에 코드가 간결하고 이해하기 쉽고, 상태를 가지지 않기 때문에 병렬 처리를 쉽게 할 수 있습니다.

- 저는 클래스형보다 함수형을 주로 사용합니다. 왜냐하면 모던자바 이후 도입된 스트림과 람다가 코드 작성시 매우 효율적이었기 때문입니다. 하지만 팀프로젝트 진행 시에는 함께 정한 컨벤션에 따라서 함수형을 사용한 경우도 있었습니다. 결과적으로 클래스형과 함수형 모두 장단점이 있기 때문에, 상황에 맞춰 적절히 사용하는 것이 중요하다고 생각합니다.

### 23. CI/CD에 대해서 설명해주세요. (8/23)

- CI/CD는 Continuous Integration, Continuous Delivery/Deployment의 약어로 지속적 통합, 지속적 제공과 배포입니다. 이러한 CI/CD는 파이프라인 형태로 구축이 되며, Build, Test, Merge, Continuous Delivery, Continuous Deployment의 순서로 진행됩니다.
- 여기서 Build, Test, Merge는 지속적 통합영역이고, 테스트를 통해 코드의 안정성을 높이는 것이 특징입니다.
- 그리고 지속적 제공은 Merge된 코드들을 배포할 브랜치로 Release하는 것을 말하며, 지속적 배포는 Release된 코드들을 프로덕션에 자동으로 배포하는 것을 말합니다.
- 결과적으로 CI/CD는 위와 같은 통합과 배포를 순서대로 정하여 자동화된 파이프라인을 구성하고 개발과 배포 과정을 효율적으로 진행하는데 도움이 됩니다.

### 24. 프로세스와 쓰레드에 대해서 설명하고 그 차이에 대해서 설명해주세요. (8/24)

- 프로세스는 컴퓨터 시스템에서 실행되는 작업의 단위이고, 쓰레드는 프로세스 내에서 실행되는 실행 흐름의 단위입니다.
- 차이점으로는 크게 독립성과 자원의 사용방식, 2가지가 있습니다.
- 첫번째로 프로세스는 독립적으로 실행되지만, 쓰레드는 프로세스 내에서 실행됩니다. 따라서 프로세스는 서로 간섭할 수 없습니다. 그러나 쓰레드는 서로 간섭할 수 있습니다.
- 두번째로 프로세스는 메모리, CPU, 파일, 장치 등 다양한 자원을 필요로 하지만 쓰레드는 프로세스가 할당받은 자원을 공유합니다. 따라서 쓰레드는 프로세스보다 자원을 효율적으로 사용할 수 있습니다.

### 25. 멀티프로세스와 멀티쓰레드의 특징에 대해 설명해주세요. (8/25)
- 멀티프로세스와 멀티쓰레드 모두 컴퓨터 시스템에서 여러 작업을 동시에 수행할 수 있도록 하는 기술입니다.
- 멀티프로세스는 각 프로세스가 CPU, 파일 등의 자원을 독립적으로 사용하기 때문에 충돌이 발생하지 않지만, 자원을 많이 사용합니다.
- 멀티쓰레드는 프로세스가 할당받은 자원을 공유하기 때문에 효율적인 자원 사용이 가능하지만, 충돌이 발생할 수 있습니다.
- 따라서 보통의 경우 CPU를 많이 사용하는 작업에는 멀티프로세스, 메모리를 많이 사용하는 작업에는 멀티쓰레드를 사용하는 것으로 알고 있습니다. 또한, 필요시 둘을 조합하여 사용한다고 알고 있습니다.

### 26. DB 로직 최소화를 하려면 어떻게 해야할까요? (8/26)
- DB로직을 최소화하는 방법은 4가지가 있습니다.
- 첫번째는 쿼리 작성 시 필요한 데이터만 조회하도록 구성하는 것입니다.
- 두번째는 데이터 검색 성능 향상을 위해 필요한 컬럼에 인덱스를 생성하는 것입니다.
- 세번째는 트랜잭션의 사용을 최소화 하는 것입니다.
- 마지막으로 네번째는 DB캐싱을 활용하는 것으로 자주 사용되는 데이터는 캐시하여 DB에 접근하는 횟수를 줄이는 것입니다.
- 이러한 DB로직의 최소화는 결과적으로 DB 쿼리의 호출 횟수를 줄여 성능이 향상되고, DB의 접근횟수를 줄이기 때문에 DB에 대한 공격 표면을 줄여 보안을 강화할 수 있습니다.

### 27. 테스트코드에 대해서 아는대로 설명해주시고 활용 경험에 대해서 답변해주세요. (8/28)
- 테스트 코드는 소프트웨어의 기능과 동작을 테스트할 때 사용하는 코드입니다.   
- 주요 사용 목적은 소프트웨어의 결함 발견, 코드 변경에 따른 부작용 예방, 코드의 품질 향상 등이 있습니다.
- 또한, 테스트 코드는 테스트 범위에 따라 단위 테스트, 통합 테스트, 기능 테스트 등으로 나뉩니다.
- 저는 프로젝트에서 기능에 대한 단위 테스트를 활용해본 경험이 있습니다. 테스트 코드를 작성할 때는 시간이 이전보다 더 걸리기도 했고, 공부하는데 시간도 들었지만, CI/CD를 도입했기 때문에 통합 부분의 안정성을 높일 수 있었고 이후 추가 코드를 작성 시 문제 확인에 도움이 되었습니다.

### 28. Array, LinkedList에 대해 설명해주시고 각각 어떻게 사용하는지 말씀해주세요.
- Array와 LinkedList는 데이터를 저장하기 위한 자료구조입니다.
- Array는 연속된 메모리 공간에 데이터를 저장하는 구조이며, Index를 사용하여 데이터에 접근할 수 있습니다. Array는 데이터의 순서가 중요하고, 데이터를 빠르게 접근해야 하는 경우에 사용합니다.
- LinkedList는 각 데이터가 이전 데이터와 다음 데이터의 주소를 가지고 있는 구조입니다. 따라서 데이터의 순서는 연결된 주소에 따라 결정됩니다. LinkedList는 데이터의 삽입 및 삭제가 빈번한 경우에 주로 사용합니다.
- 결과적으로 두 자료구조의 특징이 다르기 때문에 알맞은 상황에 사용하는 것이 중요합니다.

### 29. AWS S3, EC2를 사용하는 이유와 사용 경험에 대해서 답변해주세요.
- AWS S3와 EC2를 사용하는 이유는 트래픽 증가에 따라 인프라를 확장할 수 있고, 보안 기능을 제공하며 사용량을 기반으로 한 요금제이기 때문에 비용을 효율적으로 사용할 수 있기 때문입니다.
- 저는 AWS S3를 사용하여 이미지 업로드 기능 시 이미지를 저장한 경험이 있고, EC2를 사용하여 어플리케이션 서버를 배포한 경험이 있습니다. 두 서비스 모두 편리하게 사용할 수 있는 클라우드 컴퓨팅 서비스라고 생각합니다.

### 30. 정렬 알고리즘에 대해서 아는대로 설명해주세요.
- 정렬 알고리즘은 주어진 데이터를 특정한 기준에 따라서 순서대로 나열하는 알고리즘입니다
- 정렬 알고리즘은 크게 비교 정렬과 비교하지 않는 정렬로 나눌 수 있습니다. 비교 정렬은 데이터를 비교하여 순서를 정하는 알고리즘으로, 가장 일반적으로 사용되는 정렬 알고리즘입니다. 
- 비교하지 않는 정렬은 데이터를 비교하지 않고 순서를 정하는 알고리즘으로, 비교 정렬에 비해 빠르지만, 데이터의 특성에 따라 성능이 떨어질 수 있습니다.
- 정렬 알고리즘의 성능은 시간 복잡도와 공간 복잡도로 측정됩니다. 시간 복잡도는 정렬 알고리즘이 데이터를 정렬하는 데 걸리는 시간을 나타내는 척도입니다. 공간 복잡도는 정렬 알고리즘이 사용하는 메모리 공간의 크기를 나타내는 척도입니다.
- 정렬 알고리즘은 데이터의 특성과 요구 사항에 따라 적절한 알고리즘을 선택해야 합니다. 예를 들어, 데이터의 크기가 작다면 비교 정렬 알고리즘을 사용해도 충분합니다. 그러나 데이터의 크기가 크다면 비교하지 않는 정렬 알고리즘을 사용하면 더 효율적으로 정렬할 수 있습니다.
